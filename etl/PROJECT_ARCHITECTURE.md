# Архитектура проекта ETL

## Назначение проекта
Проект предназначен для отказоустойчивого переноса данных из PostgreSQL в Elasticsearch.


## Общая схема
```
PostgreSQL --> extract/postgres_extractor.py -->

--> transform/transformer.py --> models.py -->

--> load/elastic.py --> Elasticsearch <-- state/state.py

```

## Основные слои

### `main.py`
Точка входа приложения, которая отвечает за:
- загрузку настроек;
- настройку логирования;
- создание ETL-компонентов;
- запуск бесконечного ETL-цикла.

### `config.py`
Конфигурации приложения.
Отвечает за:
- чтение переменных окружения;
- валидацию настроек через `pydantic`;
- формирование параметров подключения к PostgreSQL.

### `extract/`
Слой извлечения данных из PostgreSQL.

Содержит:
- `postgres_extractor.py` — основную логику чтения изменений данных;
- `queries.py` — SQL-запросы.

Слой extract:
- определяет, какие сущности изменились;
- находит связанные `film_work`;
- загружает данные кинопроизведения, жанров и персон.

### `transform/`
Слой преобразования данных.

Содержит:
- `transformer.py` — функции для преобразования данных и сборки итогового документа фильма.

Слой transform:
- группирует жанры по кинопроизведению;
- группирует персон по ролям;
- собирает объект `FilmWork` для записи в Elasticsearch.

### `models.py`
Описывает структуры данных ETL.

### `load/`
Слой загрузки данных в Elasticsearch.

Содержит:
- `elastic.py` — клиент и логика загрузки;
- `es_schema.py` — схема индекса `movies`.

Слой load:
- создаёт индекс при необходимости;
- загружает документы пачками;
- использует `film_work.id` как `_id` документа в Elasticsearch.

### `state/`
Слой хранения состояния ETL.

Содержит:
- `base.py` — абстракцию хранилища состояния;
- `json_storage.py` — файловое JSON-хранилище;
- `state.py` — работу с локальной копией состояния.

Слой state:
- хранит checkpoint `(modified, id)`;
- восстанавливает состояние при старте приложения;
- обновляет состояние после успешной загрузки данных.

### `pipeline/`
Слой оркестрации.

Содержит:
- `orchestrator.py` — координацию работы ETL-компонентов.

Оркестратор:
- запускает один проход ETL по каждому типу сущности;
- берёт checkpoint `(modified, id)` из `state`;
- вызывает extract;
- разбивает результат на батчи;
- передаёт батчи в load;
- сохраняет новый checkpoint после успешной обработки.

## Поток обработки данных
ETL работает циклически:
1. читает изменения из PostgreSQL;
2. собирает итоговые документы фильмов;
3. загружает их в индекс `movies` в Elasticsearch;
4. сохраняет состояние, чтобы после перезапуска продолжить работу.

Во время одного прохода ETL выполняется следующая последовательность:
1. Оркестратор получает checkpoint для текущего типа сущности.
2. Extract читает изменившиеся записи из PostgreSQL по `(modified, id)`, определяет список затронутых кинопроизведений, загружает недостоющую информацию по произведениям, жанрам и персонам.
3. Transform собирает объекты `FilmWork`.
4. Оркестратор разбивает фильмы на батчи.
5. Load записывает батчи в Elasticsearch.
6. State сохраняет новый checkpoint `(modified, id)`.

## Отказоустойчивость
В проекте используется механизм `backoff`, что позволяет ETL не завершаться сразу при временной недоступности сервисов, а повторять попытки с увеличивающейся задержкой.
